# DECISIONS — 결정과 이유

자율 실행 규칙에 따라 모호한 사항을 직접 결정한 기록.

1. **결정: PPTX를 빌드타임 변환기(`tools/convert.js`)로 절대좌표 HTML로 자동 변환** —
   이유: 사용자 지시의 최우선 요구가 "52장 전체를 한 장도 빠짐없이 1:1 변환"이므로,
   수작업 재제작 대신 OOXML 파싱 기반 자동 변환이 누락·왜곡 위험이 가장 낮다.
   변환 결과는 `js/content.js`(콘텐츠 데이터)로 분리되어 작업지시서의 Content-Driven 원칙을 충족한다.

2. **결정: 작업지시서의 샘플 12장 구조 대신 실제 PPT 52장을 그대로 사용** —
   이유: 작업지시서(PRD)는 범용 템플릿 기준의 문서이고, 사용자 지시가 "PPT 소스 전 슬라이드 변환"으로
   더 구체적이므로 사용자 지시가 우선한다. PRD의 발표 시스템 요구(내비게이션·Overview·전환·HUD·노트)는 전부 구현했다.

3. **결정: 폰트는 나눔스퀘어 네오(TTF)를 본문에, 에이투지체(A2Z, OTF)를 HUD/목차 UI에 사용** —
   이유: 원본 PPT의 주력 폰트가 나눔스퀘어 네오 계열(3,000회 이상 사용)이다. 원본에 일부 등장하는
   Pretendard/맑은 고딕/Noto 등은 저장소에 포함되지 않아, 시각적으로 가장 가까운 나눔스퀘어 네오로 매핑했다.
   저장소 제공 폰트인 에이투지체는 발표 시스템 크롬(HUD·Overview) 전용 디스플레이 폰트로 사용해 두 폰트 모두 로컬 에셋으로 활용한다.

4. **결정: 라이브러리는 GSAP만 로컬 번들(`js/vendor/gsap.min.js`)로 포함, Three.js·Lenis는 미포함** —
   이유: 본 덱은 슬라이드형(스크롤형 아님)이라 Lenis(스무스 스크롤)는 적용 지점이 없고,
   WebGL 3D는 "발표 안정성 최우선·저사양 폴백" 원칙에 따라 Canvas 2D 파티클(1,400개)로 대체했다.
   불필요한 600KB+ 번들을 배제하는 것이 Performance Budget(초기 로딩 1~3초)에 부합한다.

5. **결정: 전환 체계 = 챕터 경계 wipe / 챕터 내 push / 히어로·클로징 zoom + 파티클** —
   이유: PRD 5.2의 "동일 계층 동일 전환, 챕터 경계 구분" 규칙을 따르되,
   Morph/Camera Fly-through 등 고위험 효과는 1:1 비즈니스 콘텐츠의 가독성·안정성을 해치지 않는 범위로 절제했다("1씬 1임팩트").

6. **결정: 임베디드 미디어(pptx 내부 161개)를 단일 소스로 사용, `/video` 폴더의 5개 mp4는 별도 복사 안 함** —
   이유: 바이트 크기 대조 결과 `/video`의 5개 파일이 pptx 임베디드 media5/9/10/12/13.mp4와 동일 파일로 확인됐다.
   중복 커밋을 피하고 슬라이드-미디어 관계(rels)를 그대로 보존한다. 로컬 에셋 사용 원칙(외부 CDN 0건)은 동일하게 충족된다.

7. **결정: 풀블리드(≥1650×920px) 영상은 muted 자동재생·루프, 그 외 영상은 controls 유지** —
   이유: 원본 PPT에서 전면 영상은 배경 연출용, 작은 영상은 데모 시연용으로 쓰였다.
   브라우저 자동재생 정책상 muted만 허용되므로 배경 영상은 음소거 루프가 표준 관행이다.

8. **결정: Overview 썸네일은 정적 캡처가 아닌 실제 슬라이드 DOM의 scale 축소 렌더링** —
   이유: PRD 5.1 구현 방식 요구("정적 이미지 캡처에만 의존하지 않는다")를 그대로 따랐다.
   썸네일 내 video 태그만 포스터 이미지로 치환해 메모리를 절약했다.

9. **결정: higgsfield MCP 이미지·영상 생성 미사용** —
   이유: 52장 1:1 변환에 필요한 모든 이미지(152개)·영상(13개)이 원본 pptx에 이미 존재해
   추가 생성이 필요한 에셋이 발생하지 않았다. (PENDING_ASSETS.md 참조)

10. **결정: 발표자 노트는 원본 pptx의 notesSlides에서 추출해 팝업 창(N 키)으로 제공** —
    이유: PRD 권장 산출물(speaker-notes)을 원본 데이터 손실 없이 구현하는 가장 단순한 방법.
    팝업 차단 시 콘솔 안내 후 발표 흐름은 계속된다(우아한 폴백).
